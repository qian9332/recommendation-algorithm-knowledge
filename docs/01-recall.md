# 一、基础召回链路设计

## 1.1 多路召回架构原理

### 核心定义

多路召回是指在召回阶段并行启用多条异构策略，每条策略基于不同特征或算法（如协同过滤、双塔向量、知识图谱、实时行为序列、规则模板等），各自从全库中截取少量候选，再通过动态配额算法融合成最终百级候选送入排序层。

### 业务价值量化

| 指标 | 单路召回 | 五路召回 | 提升幅度 |
|-----|---------|---------|---------|
| 召回率 | ~40% | ~65% | +25% |
| 长尾商品覆盖率 | 基准 | +10% | - |
| 长尾GMV占比 | 基准 | +5% | - |
| 总延迟 | - | <15ms | - |

### 国内落地案例

**淘宝搜索**：多路异构召回用i2i、u2i2i、向量、图谱、实时行为五路，配额由Contextual Bandit在线调整

**抖音推荐**：双塔+多路召回把短兴趣、长兴趣、社交、地理四路做log-q分布校准后再融合

## 1.2 倒排索引构建与更新

### 物理存储结构

```
┌─────────────────────────────────────────────────────────┐
│                    倒排索引结构                          │
├─────────────────────────────────────────────────────────┤
│  TermDict段 (FST)                                       │
│  ├── 热词常驻 HugePage (2MB大页)                        │
│  └── 冷词按 LRU 换出                                    │
├─────────────────────────────────────────────────────────┤
│  Postings段                                             │
│  ├── Block: 64KB                                        │
│  ├── Chunk: 4KB                                         │
│  └── SubChunk: 256 DocID                                │
│      ├── DocID: Delta + Variable Byte 编码              │
│      └── Freq: ZigZag + PForDelta                       │
├─────────────────────────────────────────────────────────┤
│  Skip段                                                 │
│  └── 每64个DocID插一条SkipEntry                         │
│      └── <DocID, ChunkOffset, MaxScore> (16B)           │
└─────────────────────────────────────────────────────────┘
```

### 延迟优化策略

1. **热词FST常驻HugePage**：覆盖95%查询，CPU Cache Miss下降18%
2. **超高频词转RoaringBitmap**：Seek操作变位运算，P99延迟再降120µs
3. **AVX-512解压批量化**：P99可再降15%

### 增量更新策略

日均增量500万商品时：
- Flink双流join + 5秒滚动窗口
- 版本化索引写入（双buffer热切换）
- 回滚能力：30秒内回滚到任意历史版本

## 1.3 文本语义召回技术

### BM25打分公式详解

```
Score(Q, D) = Σ_{t∈Q} IDF(t) · [tf(t,D)·(k1+1)] / [tf(t,D)+k1·(1-b+b·|D|/avgdl)]
```

**参数物理意义**：

| 参数 | 含义 | 典型取值 |
|-----|------|---------|
| k1 | 词频饱和拐点 | 1.2~2.0（电商标题） |
| b | 文档长度惩罚系数 | 0.75（商品搜索） |
| IDF(t) | 词重要性 | log[(N−df(t)+0.5)/(df(t)+0.5)] |

### 双塔DSSM负样本构造

#### 四种策略对比

| 策略 | 实现复杂度 | 效果 | 适用场景 |
|-----|-----------|------|---------|
| 随机负采样 | 低 | 基准 | 快速验证 |
| Batch内负采样 | 低 | +1.5% AUC | 大规模训练 |
| 曝光未点击 | 中 | +2% AUC | 精细化优化 |
| 难负例挖掘 | 高 | +3% AUC | 效果优先 |

#### 实战组合策略

```
训练步前 20%：用 INC 打基础分布
中间 60%：用 In-Batch 提速
最后 20%：插入难负例做精细化拉升
随机负例作为正则，比例控制在 5~10%
```

## 1.4 实时用户行为触发召回

### 30分钟点击序列实时I2I召回

```
┌─────────────────────────────────────────────────────────┐
│                   实时I2I召回架构                        │
├─────────────────────────────────────────────────────────┤
│  数据采集层                                             │
│  └── Flink消费Kafka click流，维护30min Item ID队列      │
├─────────────────────────────────────────────────────────┤
│  向量生成层                                             │
│  ├── Item塔：每天凌晨训练，128维向量存Tair-Vector      │
│  └── 用户实时兴趣塔：时间衰减Attention，α=0.9           │
├─────────────────────────────────────────────────────────┤
│  在线召回层                                             │
│  ├── Tair-Vector HNSW接口，Top-500 10ms返回            │
│  ├── 规则过滤（库存>0、价格区间、曝光去重）             │
│  └── 轻量GBDT（40棵树）输出50个商品给粗排               │
└─────────────────────────────────────────────────────────┘
```

### 防止误点击抖动

1. **埋点端**：有效点击定义为停留≥800ms且页面可见度≥50%
2. **Flink CEP**：5秒滑动窗口，同item连续点击≥3次且间隔<300ms只保留第一次
3. **训练侧**：噪声权重乘到loss上，对比学习约束
4. **在线降级**：实时向量与上一版本余弦距离>0.15时自动降级

## 1.5 召回效果评估

### 增量召回率计算

```
ΔRecall = Recall_sys(with new) – Recall_sys(without)

# 必须做"去重"，防止其它路已经把样本召回
# Shadow实验：新召回只记录不打乱原队列
```

### 评估指标体系

| 指标 | 计算方式 | 目标值 |
|-----|---------|--------|
| Recall@K | 用户转化集合被召回的比例 | >65% |
| Precision@K | 召回结果中有效商品比例 | >8% |
