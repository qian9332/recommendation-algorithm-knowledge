# 十五、推荐后端工程师岗位全景指南

> 以顶级推荐后端工程师视角，系统梳理岗位职责、日常工作、核心技术、工作流程。

---

## 15.1 岗位职责概述

### 15.1.1 核心定位

```
推荐后端工程师的核心定位：

┌─────────────────────────────────────────────────────────────┐
│                    推荐后端工程师                            │
├─────────────────────────────────────────────────────────────┤
│  职责：构建高可用、高性能、可扩展的推荐系统基础设施          │
│                                                             │
│  核心价值：                                                  │
│  ├── 让算法模型能够稳定、高效地在线上运行                    │
│  ├── 保证系统在高并发下的稳定性和低延迟                      │
│  ├── 提供灵活的特征服务和模型更新能力                        │
│  └── 支撑业务快速迭代和算法持续优化                          │
└─────────────────────────────────────────────────────────────┘
```

### 15.1.2 职责边界

```
推荐后端工程师 vs 算法工程师：

┌─────────────────┬─────────────────────┬─────────────────────┐
│ 维度            │ 推荐后端工程师      │ 算法工程师          │
├─────────────────┼─────────────────────┼─────────────────────┤
│ 核心关注        │ 系统稳定性、性能    │ 模型效果、精度      │
│ 主要产出        │ 在线服务、特征平台  │ 模型、特征策略      │
│ 技术栈          │ 分布式系统、高并发  │ 机器学习、深度学习  │
│ 成功指标        │ QPS、延迟、可用性   │ AUC、CTR、GMV       │
│ 日常工作        │ 系统优化、故障处理  │ 模型训练、效果分析  │
└─────────────────┴─────────────────────┴─────────────────────┘

协作模式：
├── 算法工程师负责模型效果优化
├── 后端工程师负责模型在线部署和性能优化
├── 共同负责特征工程和A/B实验
└── 共同负责线上问题排查
```

## 15.2 日常工作内容

### 15.2.1 日常工作清单

```
推荐后端工程师日常工作清单：

【每日必做】
├── 09:00-09:30 查看系统监控
│   ├── 检查P99延迟是否正常
│   ├── 检查QPS波动
│   ├── 检查错误率
│   └── 查看资源使用率
│
├── 09:30-10:00 处理告警和工单
│   ├── 响应夜间告警
│   ├── 处理业务方反馈
│   └── 跟进未解决问题
│
├── 10:00-12:00 核心开发工作
│   ├── 需求开发
│   ├── 性能优化
│   ├── 代码评审
│   └── 技术方案设计
│
├── 14:00-17:00 系统优化和维护
│   ├── 热点问题排查
│   ├── 容量规划
│   ├── 灰度发布
│   └── A/B实验支持
│
├── 17:00-18:00 文档和总结
│   ├── 技术文档编写
│   ├── 问题复盘
│   └── 周报/日报
│
└── 随时待命
    ├── 线上故障响应
    ├── 紧急需求支持
    └── 跨团队协作

├── 周会：同步项目进度、风险
├── 技术分享：团队内部技术交流

├── 系统巡检：全面检查系统健康度
├── 成本优化：评估资源使用效率
├── 技术规划：制定下月技术计划
└── 故障复盘：总结本月故障和改进
```

### 15.2.2 工作优先级

```
工作优先级排序：

P0 - 立即处理（影响线上服务）
├── 线上故障
├── 服务不可用
├── 数据丢失风险
└── 安全漏洞

P1 - 当天处理（影响业务）
├── 性能严重下降
├── 核心功能异常
├── 重要业务需求
└── 关键告警

P2 - 本周处理（重要不紧急）
├── 性能优化
├── 代码重构
├── 技术债务
└── 文档完善

P3 - 排期处理（优化类）
├── 体验优化
├── 工具开发
├── 技术调研
└── 知识分享
```

### 15.4.2 故障处理流程

```

【故障发现】
├── 监控告警
├── 用户反馈
├── 业务方反馈
└── 巡检发现

【故障响应】（5分钟内）
├── 确认故障
│   ├── 影响范围
│   ├── 严重程度
│   └── 业务影响

├── 建立沟通
│   ├── 创建故障群
│   ├── 拉入相关人员

└── 初步止损
    ├── 快速回滚
    ├── 降级处理
    └── 流量切换

【故障定位】（30分钟内）
├── 收集信息
│   ├── 日志分析
│   ├── 监控分析
│   ├── 变更记录
│   └── 依赖状态
│
├── 分析原因
│   ├── 代码问题
│   ├── 配置问题
│   ├── 资源问题
│   ├── 依赖问题
│   └── 流量问题
│
└── 验证假设
    ├── 复现问题
    ├── 修复验证
    └── 灰度验证

【故障修复】
├── 制定修复方案
├── 代码修复
├── 测试验证
├── 发布上线
└── 验证效果

【故障复盘】（24小时内）
├── 故障时间线
├── 影响范围

├── 处理过程
├── 经验教训

```
